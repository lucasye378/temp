# DECISION-PACK-REALISH-VALIDATION-UNIT-V1.md

## Goal

设计一次更接近真实交付闭环的“准真实验证单元”，
让决策包服务不只停在样例展示，
而开始具备：

- 明确场景
- 明确交付
- 明确最小反馈问题
- 明确通过 / 不通过标准

---

## 1. Chosen validation unit

### 验证题目
**写作 + 研究场景下，ChatGPT / Claude / Gemini 的第一次选择决策包**

这次不重新发明题目，
直接使用已经完成的样例：
- `AI-TOOLS-DECISION-PACK-SAMPLE-V1.md`

原因：
- 场景足够具体
- 候选范围清楚
- 已经有初步建议与下一步动作
- 最容易接入反馈采集

---

## 2. Delivery unit

这次准真实验证单元的交付物由两部分组成：

### A. 决策包本体
- 直接交付 `AI-TOOLS-DECISION-PACK-SAMPLE-V1.md`

### B. 最小反馈采集
交付后立刻配套 3 个问题：

1. **看完后，你现在是否更容易做第一次选择了？**
   - 是 / 一般 / 否

2. **这份包有没有帮你排除掉至少 1 个候选？**
   - 有 / 没有

3. **你接下来最可能马上做的动作是什么？**
   - 先试 Claude / 先试 ChatGPT / 先试 Gemini / 还需要再想想

可选补第 4 问：

4. **这份包相比你自己查资料，省事多少？**
   - 明显省事 / 有一点 / 没感觉

---

## 3. Why this counts as “realish” instead of just sample

因为它已经开始具备验证闭环最小结构：

- 有明确用户场景
- 有实际交付物
- 有明确反馈问题
- 有可判断的通过标准

它还不是完全真实交付，
因为：
- 还没有真实用户回收结果
- 还没有基于结果改第二版

但它已经比“单纯展示样例”更接近真实验证。

---

## 4. Pass / fail rule

如果后续反馈命中以下 3 条及以上，就算这次准真实验证成功：

1. 用户明确更容易做第一次选择
2. 用户排除了至少 1 个候选
3. 用户能选出一个首试项
4. 用户主观觉得明显更省事
5. 用户愿意继续要第二份类似决策包

如果没命中 3 条：
- 优先检查场景是否太宽
- 检查推荐逻辑是否不够锋利
- 检查下一步动作是否不够可执行

---

而是二选一：

## One-Sentence Rule

**准真实验证单元的最小标准，不是只有样例，而是“样例 + 反馈问题 + 成功判断标准”三件套同时存在。**
