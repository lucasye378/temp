# DECISION-PACK-TARGET-SOURCE-AND-MESSAGE-V1.md

## Goal

把“决策包服务”第一次真实反馈验证的下一步继续压实：

- 第一批对象可能从哪里来
- 第一条发送文案该怎么写

这一步仍然不是实际外发，
而是把后续外发前最容易卡住的部分先定义清楚。

---

## 1. Object source types v1

第一轮对象来源，不追求多，
只追求：
- 问题真实存在
- 场景对题
- 愿意回复

### Source A — 已经明确聊过 AI 工具选择的人
最优先。

原因：
- 问题真实存在
- 不需要重新教育场景
- 更容易得到有效反馈

### Source B — 最近确实在做写作 / 阅读 / 研究任务的人
次优先。

原因：
- 场景匹配度高
- 看完更可能立刻试用

### Source C — 愿意配合做产品验证的人
可作为补充。

原因：
- 反馈意愿高
- 但如果问题不够真实，反馈含金量会下降

---

## 2. Source priority rule

如果同时有多个候选对象，
优先级应是：

1. **问题真实存在**
2. **场景匹配**
3. **愿意给反馈**
4. **容易联系**

注意：

**“容易联系”不能排在前面。**

---

## 3. First message skeleton v1

第一条发送文案要完成 4 件事：

1. 说明为什么发给他
2. 说明这不是群发资料
3. 说明希望他给最小反馈
4. 不把信息写得过长

---

## 4. First message draft v1

可用骨架：

> 我做了一份很小的决策包，主题是：
> “写作 + 研究场景下，第一次先试 ChatGPT / Claude / Gemini 哪一个更合适？”
>
> 我想到你，是因为你比较像这个问题真的存在的人。
>
> 如果你愿意，我发你这份很短的版本。
> 你看完后只要告诉我：
> - 现在是不是更容易做第一次选择了
> - 有没有帮你排除掉至少 1 个候选
> - 你接下来最可能先试哪个
>
> 不用长反馈，几句就够。

---

## 5. Why this draft works

这版骨架有几个优点：

- 先说明“为什么想到你”，不是生硬丢文件
- 明确这是一个小验证，不是假装正式推销
- 反馈成本低，不会让人觉得麻烦
- 先问是否愿意，再发正文，更尊重边界

---

## 6. Recommended next task

下一步最合理的是：

**基于这份对象来源与首条文案骨架，判断当前是否已经具备“等待明确对象后即可外发”的条件。**

---

## One-Sentence Rule

**第一条发送文案的目标，不是把包内容提前讲完，而是让对方愿意进入“看包 + 给最小反馈”这条链。**
