# DECISION-PACK-VALIDATION-SCENARIO-V1.md

## Goal

把“第一轮研究 / 决策包服务”从内部样例，进一步压缩成一个**可直接对外交付、可被真实用户理解、也方便验证价值**的首个场景。

这一步不是做完整商业化方案，而是定义一个：
- 边界清楚
- 价值容易感知
- Allen 现在就能交付
- 可以快速复制第二单的验证场景

## Chosen Validation Scenario

### 场景名称
**AI 工具选择决策包 v1**

### 一句话定义
当一个个人用户或小团队面对 3–5 个 AI 工具，不知道该先用哪一个时，Allen 交付一份“第一轮选择决策包”，帮助对方在短时间内做出更稳的初步选择。

## Why this scenario wins

相比“项目方向比较”或“购买决策研究”，这个场景更适合作为首个外部化验证样例，原因有 5 个：

1. **问题非常真实且高频**
   - 现在每个人都在面对 AI 工具过载：ChatGPT、Claude、Gemini、Perplexity、Notion AI、Cursor、Copilot……
   - 不是没人有需求，而是很多人已经被选择成本困住。

2. **价值容易立刻感知**
   - 用户很容易明白：这份包是在帮他省研究时间、减少试错、快速开始。
   - 价值不是抽象“咨询”，而是“告诉我先用哪个、为什么”。

3. **边界比通用研究更容易收窄**
   - 工具选择天然可以限制在某个任务场景里，比如：写作、研究、编程、团队知识管理。
   - 不容易无限发散成“大而全研究”。

4. **Allen 现在就能做**
   - 需要的核心能力是：信息整理、差异比较、场景匹配、结构化建议。
   - 不依赖复杂开发，不依赖私有数据，不依赖长周期执行。

5. **容易扩展到相邻场景**
   - AI 工具选型做顺了，后面很自然可以扩展到：软件方案对比、工作流工具选择、服务商初筛、购买决策研究。

## Target User v1

### 最适合的首批对象
- 想开始系统用 AI，但不知道从哪一个工具下手的个人用户
- 需要为一个具体工作场景选 AI 工具的小团队
- 对工具很多、信息很杂、没空逐个试用的人

### 暂不优先的对象
- 超大型企业采购
- 需要严格 security / compliance 审计的正式选型
- 要求完整实施落地和长期陪跑的客户

因为这些场景会把服务边界迅速拉大，不适合首个验证样例。

## User Problem Statement

用户的真实困境通常不是“完全不知道有哪些工具”，而是：
- 看过很多推荐，但越看越乱
- 每个工具都说自己好，很难比较
- 不知道什么适合自己的具体场景
- 害怕选错后浪费时间迁移
- 没空自己做第一轮研究

换句话说，用户缺的不是信息，而是：
**第一轮筛选、场景化比较、以及一个足够可信的起步建议。**

## Deliverable v1

首个“AI 工具选择决策包 v1”建议固定为以下结构：

### 1. 用户目标重述
- 用户要解决的核心任务是什么
- 有哪些约束（预算、易用性、中文支持、团队协作、代码能力等）

### 2. 候选工具列表（3–5 个）
- 不追求全，而是只保留值得比较的候选

### 3. 对比矩阵
每个候选至少比较：
- 最适合的使用场景
- 主要优势
- 主要短板
- 成本 / 上手门槛
- 不适合谁

### 4. 初步结论
- 若只先试 1 个，推荐哪个
- 若保留 2 个备选，分别是什么
- 为什么不是另一个

### 5. 风险提醒
- 哪些判断目前仍有不确定性
- 哪些最好通过 1–3 天的轻量试用验证

### 6. 下一步动作
给出极简行动建议，例如：
- 今天先注册哪个工具
- 先跑哪 2 个真实任务
- 用什么标准判断要不要继续用

## Example Request Forms

这个样例最适合的请求长这样：

- “我主要做写作和研究，ChatGPT / Claude / Gemini 先用哪个？”
- “我们 3 人小团队想上 AI，Notion AI、ChatGPT Team、Claude 哪个更适合先试？”
- “我是程序员，Cursor、Copilot、Claude Code 类工具先从哪个开始最值？”

## Value Proposition

这个服务卖的不是“百科全书式工具介绍”，而是：

### 节省的东西
- 节省用户自己查资料和横向比较的时间
- 节省错误试用和迁移的成本
- 节省“迟迟不开始”的决策摩擦

### 提供的东西
- 第一轮比较
- 初步判断
- 更稳的起步路径

## Service Boundary v1

为了避免失控，首版明确不做：
- 超过 5 个候选的大范围研究
- 深度技术测试 / benchmark
- 企业级采购方案
- 长期实施与培训
- 需要持续跟踪更新的重度顾问服务

### 边界原则
首版只解决一件事：
**帮用户完成第一次更稳的选择。**

## Success Criteria

如果这个场景要算验证成功，至少应出现以下信号中的 2–3 个：

1. 用户能一眼看懂这是在帮他解决什么问题
2. 用户收到后，能据此立即开始试一个工具
3. 用户觉得这比自己查半天更省事
4. 用户愿意继续要第二份类似决策包
5. 用户愿意为更复杂版本付费或转介绍

## Why not the other candidate scenarios first

### 不先做“项目方向比较包”
- 它适合作为内部能力样例，但对外理解门槛稍高
- 用户群体没有“AI 工具选择”那么广、那么即时

### 不先做“购买决策研究包”
- 虽然价值直观，但容易陷入价格、渠道、库存、参数细节泥潭
- 边界控制比 AI 工具场景更难

### 不先做“通用软件方案对比包”
- 对外虽有价值，但不同软件类别差异太大
- 作为首个场景，AI 工具更热、更容易被用户立即感知
