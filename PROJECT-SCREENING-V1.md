# PROJECT-SCREENING-V1.md

## Goal

把项目探索从“零散想法池”升级成“可比较、可筛选、可决定是否进入验证”的机制。

这版 v1 不追求完美，只追求：
- 能稳定筛第一轮
- 能压缩选择
- 能逼出下一个真实验证动作

## Scope

适用于当前阶段的项目候选，尤其是：
- 低编程门槛
- 可先人工 / 半自动交付
- 能较快验证现实价值
- 能同时服务 Allen 的成长与 Goal 2

## Screening Flow v1

### Step 1 — 入池条件

一个方向进入候选池，至少要满足以下 3 条中的 2 条：
1. 指向一个真实存在的低效问题
2. Allen 现在就能介入并产出第一版结果
3. 用户能在短时间内感知到价值

若连这一层都过不了，只保留在想法备忘，不进入正式候选池。

### Step 2 — 第一轮筛选（Pass / Hold / Drop）

每个候选方向先过 6 个问题：

1. **问题是否真实且高频？**
   - 是否不是“也许有人会需要”，而是现实里反复出现的问题。

2. **Allen 是否现在就能做第一版？**
   - 不靠重开发，不等复杂外部依赖。

3. **价值是否容易被用户直接感知？**
   - 用户是否能很快理解“这替我省了什么”。

4. **能否在 1–3 个运行单元内做出验证样例？**
   - 如果连样例都迟迟做不出来，先不应抢占主链。

5. **边界是否容易收窄？**
   - 若很容易无限膨胀、失控，就先降级。

6. **是否同时推动 Goal 1 与 Goal 2？**
   - 不只是赚钱想象，也不只是抽象成长；应同时帮助 Allen 提升现实执行能力，并逼近有价值项目。

### Step 3 — 打分（1–5 分）

每个候选按下列 5 个维度打分：

- 问题强度
- 立即可执行性
- 用户感知价值
- 验证速度
- 与 Allen 当前能力匹配度

总分解释：
- **22–25**：可优先进入验证候选
- **18–21**：保留为强候选，等更好时机
- **14–17**：先 incubate，不进主链
- **≤13**：当前阶段不做

## Decision Rules

### 何时进入主链 active

一个方向只有在同时满足以下条件时，才可从候选升级为 active 验证任务：
- 已有第一轮比较结果
- 已明确“为什么先做它，而不是另一个”
- 已定义一个最小验证样例
- 已有清楚的下一步交付物

### 何时只留在 incubating

若出现以下任一情况，则不进主链：
- 价值判断仍然太抽象
- 范围过大，无法收窄成样例
- 太依赖编程 / 外部资源 / 用户数据
- Allen 当前虽感兴趣，但无法在短周期内逼出结果

## Current Judgment

基于 v1 筛选标准：

### 当前最值得优先推进的 2 个近邻方向
1. **第一轮研究 / 决策包服务**
2. **复杂信息整理与行动化服务**

其中：
- 前者更适合作为当前主验证方向
- 后者可作为后续扩展能力或邻近服务方向

而“服务 / 工具方案对比服务”本质上可以视为“决策包服务”的一个具体应用场景，不必单独拆成主项目。

## Forced Output Rule

筛选机制的目的不是继续囤积想法，而是每轮必须逼出一个明确结论：
- **继续验证哪个方向**
- **暂停哪个方向**
- **下一份样例要做什么**

如果一轮筛选结束后没有产生这 3 个输出之一，说明这一轮筛选没有完成。

## Next Recommended Action

基于这版机制，下一步应把当前主线从“项目方向筛选”推进到：

### 为“第一轮研究 / 决策包服务”定义更窄的首个外部化验证场景 v1

候选场景示例：
- AI 工具选择包
- 小团队软件方案对比包
- 某类购买决策研究包
- 某个项目方向比较包

目标不是再讨论方向，而是把方向压缩成一个可直接交付的验证样例。
