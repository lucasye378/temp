# CHAIN-RULES.md

## Direct Continue vs Timer Continue — v1

### Purpose

这份文件定义 Allen 在一个运行单元结束后，应该：
- 直接开始下一个任务
- 还是通过短计时器 / 调度点重新吊起 main

目标不是追求极限速度，而是保证：
- 链条不断
- 系统稳定
- 可监控
- 可纠偏
- 不因连续运转过长而失控

## Core Principle

### Rule 1 — 当前单元的收尾必须立即完成

一个运行单元结束后，不应先跳走。
必须立刻完成以下收尾动作：
- 记录本单元结果
- 扫描新目标 / 新问题 / 新风险
- 将应吸纳的内容写入 `Todo.md`
- 生成 follow-up 候选

这些动作属于当前单元的一部分，不应延后。

## Continue Modes

### Mode A — 直接接续（Direct Continue）

适用于：
- 下一个动作非常明确
- 风险低
- 不需要重新查看全局优先级
- 预计耗时很短（建议 <= 5 分钟）
- 本质上仍属于当前工作链条的自然延伸

典型例子：
- 补一条记录
- 更新一个 Todo
- 做一次短验证
- 完成一个紧接着的收尾动作

### Mode B — 定时器接续（Timer Continue）

适用于：
- 下一步已经构成一个新的完整运行单元
- 需要重新查看主目标 / Todo / 优先级
- 需要一个明显监控断点
- 需要避免系统连续失控地一直跑下去
- 需要给 Lucas 留出观察和插话空间

建议定时器时长：
- 30 秒到 2 分钟之间
- 默认优先 30 秒或 60 秒，除非有明确理由更长

## Decision Rules

### Rule 2 — 先收尾，再判断接续方式

不要在单元尚未收尾时决定继续方式。
判断 Direct / Timer 的前提，是当前单元的记录、Todo 吸纳、follow-up 生成已经完成。

### Rule 3 — 短、清晰、低风险，才允许 Direct Continue

只有同时满足下列条件时，才应直接接续：
- 目标明确
- 边界明确
- 风险低
- 预计耗时短
- 不需要重新排序全局任务

缺一项，就优先走 Timer Continue。

### Rule 4 — 新一轮调度默认走 Timer Continue

如果当前任务结束后，需要在多个 follow-up 候选里重新选择主任务，则默认进入定时器接续，而不是立即继续。

### Rule 5 — 需要 Lucas 观察或可能纠偏时，优先 Timer Continue

如果下一步很可能需要 Lucas：
- 观察
- 插话
- 修正方向
- 改优先级

则应优先使用 Timer Continue，让链条在可见断点处继续。

## Continuous Run Policy

### Rule 6 — 取消“连续子任务最多 10 个”的硬上限

旧规则中的“连续子任务最多 10 个”最初是为避免在没有稳定定时接续时失控。
现在主链的默认接续方式已经改为：**每个运行单元结束后都设置下一次 follow-up cron**。

因此：
- 不再使用“10 个子任务上限”作为硬性打断条件
- 是否 reorg，取决于状态是否混乱、主线是否失效、或 queue/follow-up 是否发生冲突
- 只要链条清晰、状态可维护，就可以持续运行

### Rule 7 — 每个运行单元结束后，都必须设置下一次 cron

无论当前单元是否顺利完成，只要主链仍需继续：
- 先完成本单元收尾
- 更新相关 md
- **先获取当前本地时间（America/New_York）**
- 再设置下一次 follow-up cron

这条规则是新的默认接续机制。
即使存在需要人工等待的外部边界，也不要把 main 停在等待态；应先把被阻塞事项降回 queue/todo，再为新的 active 挂下一次唤醒。

### Rule 8 — 不要手算时区，尤其不要忽略夏令时

设置 cron 时：
- 先确认当前本地时间
- 再推算 30–60 秒后的本地目标时间
- 最后再写入 cron 的绝对时间

如果没有先确认当前时间，就不要直接凭感觉写 UTC 时间。

### Rule 9 — 短 follow-up cron 一律程序化计算，不手写

对于 30–60 秒这种短 follow-up：
- 不要手写目标 UTC 时间
- 先取 `now`
- 用程序算出 `now + 30s` 或 `now + 60s`
- 直接使用程序输出结果设 cron

这样可以避免“规则知道了，但最后一步还是手滑设成一小时后”的执行错误。

## Practical Recommendation (v2)

默认执行顺序：
1. 完成当前运行单元
2. 立即完成收尾
3. 更新 status / queue / memory
4. 判断下一单元的主任务
5. 设置下一次 follow-up cron
6. 由下一次唤醒继续主链

## Summary

当前阶段推荐的链条策略是：
- **单元内立即收尾**
- **单元结束后统一设置下一次 cron**
- **是否 reorg 由状态复杂度决定，而不是由固定计数决定**
- **主链应在可追踪的 cron 节点之间持续运转**
