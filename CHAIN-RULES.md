# CHAIN-RULES.md

## Direct Continue vs Timer Continue — v1

### Purpose

这份文件定义 Allen 在一个运行单元结束后，应该：
- 直接开始下一个任务
- 还是通过短计时器 / 调度点重新吊起 main

目标不是追求极限速度，而是保证：
- 链条不断
- 系统稳定
- 可监控
- 可纠偏
- 不因连续运转过长而失控

## Core Principle

### Rule 1 — 当前单元的收尾必须立即完成

一个运行单元结束后，不应先跳走。
必须立刻完成以下收尾动作：
- 记录本单元结果
- 扫描新目标 / 新问题 / 新风险
- 将应吸纳的内容写入 `Todo.md`
- 生成 follow-up 候选

这些动作属于当前单元的一部分，不应延后。

## Continue Modes

### Mode A — 直接接续（Direct Continue）

适用于：
- 下一个动作非常明确
- 风险低
- 不需要重新查看全局优先级
- 预计耗时很短（建议 <= 5 分钟）
- 本质上仍属于当前工作链条的自然延伸

典型例子：
- 补一条记录
- 更新一个 Todo
- 做一次短验证
- 完成一个紧接着的收尾动作

### Mode B — 定时器接续（Timer Continue）

适用于：
- 下一步已经构成一个新的完整运行单元
- 需要重新查看主目标 / Todo / 优先级
- 需要一个明显监控断点
- 需要避免系统连续失控地一直跑下去
- 需要给 Lucas 留出观察和插话空间

建议定时器时长：
- 30 秒到 2 分钟之间
- 默认优先 30 秒或 60 秒，除非有明确理由更长

## Decision Rules

### Rule 2 — 先收尾，再判断接续方式

不要在单元尚未收尾时决定继续方式。
判断 Direct / Timer 的前提，是当前单元的记录、Todo 吸纳、follow-up 生成已经完成。

### Rule 3 — 短、清晰、低风险，才允许 Direct Continue

只有同时满足下列条件时，才应直接接续：
- 目标明确
- 边界明确
- 风险低
- 预计耗时短
- 不需要重新排序全局任务

缺一项，就优先走 Timer Continue。

### Rule 4 — 新一轮调度默认走 Timer Continue

如果当前任务结束后，需要在多个 follow-up 候选里重新选择主任务，则默认进入定时器接续，而不是立即继续。

### Rule 5 — 需要 Lucas 观察或可能纠偏时，优先 Timer Continue

如果下一步很可能需要 Lucas：
- 观察
- 插话
- 修正方向
- 改优先级

则应优先使用 Timer Continue，让链条在可见断点处继续。

## Continuous Run Limit

### Rule 6 — 连续运转不能过多

为了避免链条无限延伸、上下文疲劳和系统失控，Allen 不应无限连续推进。

当前规则：
- **连续子任务不得超过 10 个**。

这里的“连续子任务”指：
- 在同一条连续主链中
- 没有明显重置 / 回顾 / 调度断点
- 一个接一个被吊起的连续任务单元

### Rule 7 — 达到上限后必须强制进入断点

当连续主链达到 10 个子任务时，必须停止直接续跑，并进入一个明确断点。
断点时必须完成：
- 阶段性结果总结
- Todo 整理
- 主线重新确认
- 下一阶段任务选择

也就是说：
- 10 个以内可以连续推进
- 到 10 个时必须进行一次“调度重整”

## Practical Recommendation (v1)

默认执行顺序：
1. 完成当前运行单元
2. 立即完成收尾
3. 判断 next task 是否满足 Direct Continue 条件
4. 若满足，则直接继续
5. 若不满足，则设短计时器重新吊起 main
6. 若连续子任务达到 10 个，则强制进入断点，不再直接续跑

## Summary

当前阶段推荐的链条策略是：
- **单元内立即收尾**
- **单元间分级接续**
- **连续运转上限为 10 个子任务**
- **超过上限必须强制重整主链**
